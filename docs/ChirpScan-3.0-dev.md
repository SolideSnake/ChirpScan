# ChirpScan 3.0 方向设计说明

## 1. 文档定位

本文档用于明确 ChirpScan 3.0 的产品方向、核心目标、设计边界与当前版本建议。

本文档不用于锁定具体技术框架、数据库结构、API 细节、页面细节或完整开发排期。

- 这份文档主要给项目维护者和后续开发者看，用来统一 3.0 要解决什么问题。
- 这份文档希望帮助判断 3.0 的第一版应该做什么、暂时不做什么。
- 具体模块拆分、数据库表结构、接口路径、页面布局和任务排期不写进本文档。

## 2. 背景与问题

ChirpScan 2.x 已经能够监控指定 X 账号，抓取新推文，按关键词规则过滤，并推送到 Telegram、飞书或币安广场等平台。

2.x 的主流程适合“发现新推文并通知”，但不适合长期保存、复盘和判断一条推文后续是否被验证。很多有价值内容只停留在即时通知里，看过之后很难回头检索、整理和跟踪。

3.0 要解决的是“内容沉淀”和“价值筛选”问题，而不是重写 2.x 的采集与推送系统。

- 2.x 当前已经解决了监控、采集、过滤、推送和基础状态管理。
- 当前遇到的新问题是重要推文缺少长期记录，观点缺少结构化提炼，后续验证成本较高。
- 这些问题值得进入 3.0，因为它们已经超出普通通知工具的范围，更接近“观点归档与复盘工具”。
- 不应该继续塞进 2.x 主流程，因为 2.x 应继续保持稳定、轻量和可排查。

## 3. 核心目标

只保留 3 到 5 个目标。每个目标尽量说明“要带来什么结果”，不要提前写具体实现方案。

### 目标 1：保存推文

- 想解决的问题：日常看见的推文无法保存。
- 希望带来的结果：当我通过飞书或者电报这些已有的推送渠道，发送推文链接，软件可保存原文。这个功能前端专门一页叫做“荣辱柱”
- 判断这个目标是否达成的方式：飞书电报发送的推文链接，本地界面中找到和阅读原文。

### 目标 2：让 AI 辅助提炼观点

- 想解决的问题：目标1保存的推文有太多没用的文字。
- 希望带来的结果：系统可以辅助生成结构化的观点，方便查阅。
- 判断这个目标是否达成的方式：归档内容能够产出可读、可编辑、可追溯的结构化观点，而不是只返回一段泛泛总结。

### 目标 3：从重点账号中筛出更有价值的信息

- 想解决的问题：监控的目标有很多的日常水文，希望自动判断有价值的文字发送。
- 希望带来的结果：在目前只能关键词筛选的基础上增加能帮助识别更值得阅读的推文，并且简化提炼观点。
- 判断这个目标是否达成的方式：用户能从重点账号集合中看到经过筛选的高价值内容，而不是完整信息流。

### 目标 4：爬虫监控目标的所有推文。

- 想解决的问题：有些博主要删除推文，以后无法看了。
- 希望带来的结果：能选择爬取目标的所有推文，
- 判断这个目标是否达成的方式：本地是否能阅读查阅目标的所有推文。
